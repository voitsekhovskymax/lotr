# 📜 LOTR Mod — Правила гри для AI-агентів (AGENTS.md)
## Vintage Story v1.22 | .NET 10 | C# 13

Цей документ визначає стандарти проектування, архітектурні патерни, правила написання коду та кращі практики для розробки моду **Lord of the Rings** для Vintage Story. Всі агенти та розробники повинні суворо дотримуватися цих правил для забезпечення чистоти, стабільності та продуктивності коду.

---

## 🏛️ 1. АРХІТЕКТУРНІ ПАТЕРНИ ТА СТРУКТУРА

### 🔌 1.1. Модульна архітектура систем (Mod Systems)
Вся бізнес-логіка повинна розбиватися на незалежні системи, успадковані від ванільного `ModSystem` або керовані окремими класами-сервісами.
- **Точка входу**: `LotrModSystem.cs` ініціалізує та реєструє компоненти, але не повинна містити важку логіку конкретних систем.
- **Розподіл сервісів**: Створюйте системи у папці `src/Systems/[SystemName]/` (наприклад, `AlignmentSystem.cs`, `QuestSystem.cs`).
- **Dependency Injection**: Передавайте `ICoreAPI`, `ICoreServerAPI` або `ICoreClientAPI` через конструктори сервісів. 
  - ⚠️ **Критичне правило**: Ніколи не зберігайте `ICoreAPI` у статичних полях. Це призводить до витоків пам'яті та конфліктів контексту.

### 🌐 1.2. Розділення клієнта та сервера (Client-Server Separation)
Vintage Story працює за клієнт-серверною архітектурою. Недотримання розділення сторін викликає краші гри.
- **Перевірка сторони**: Завжди перевіряйте `api.Side` перед виконанням side-specific логіки.
  - `api.Side == EnumAppSide.Server` (Сервер: бази даних, збереження, ШІ, логіка квестів).
  - `api.Side == EnumAppSide.Client` (Клієнт: рендеринг, UI, звуки, ефекти часток).
- **Відокремлення класів**: Серверний код не повинен посилатися на класи з простору імен клієнта (наприклад, `GuiScreen` або `MeshRef`) і навпаки.
- **Реєстрація сутностей**: Сутності та блоки повинні реєструватися на обох сторонах у загальному методі `Start(ICoreAPI api)`.

### ⚡ 1.3. Подієво-орієнтована архітектура (Event-Driven)
Замість постійного опитування (polling) стану гравців або світу, використовуйте подієву модель Vintage Story API:
- Підписуйтеся на події через `api.Event` (наприклад, `PlayerJoin`, `PlayerDeath`, `OnEntitySpawn`).
- Звільняйте підписки та відписуйтеся від подій при завершенні життєвого циклу моду в `Dispose()`.

## 💻 2. ПРАВИЛА КОДУВАННЯ (HOW TO CODE)

### 🚀 2.1. Асинхронне програмування (C# Async)
Дотримуйтеся найкращих практик асинхронності для запобігання фризів ігрового потоку (Main Thread):
- **Суфікс**: Всі асинхронні методи повинні мати суфікс `Async` (наприклад, `SaveDataAsync()`).
- **Повернення**: Повертайте `Task` або `Task<T>`, уникайте повернення `void` (окрім обробників подій). Для гарячих шляхів (hot paths) використовуйте `ValueTask<T>`.
- **Запобігання блокуванню**: Ніколи не використовуйте `.Wait()`, `.Result` або `.GetAwaiter().GetResult()` в асинхронному коді. Використовуйте `await`.
- **Контекст**: Використовуйте `.ConfigureAwait(false)` у бібліотечному та утилітарному коді, де не потрібен доступ до головного ігрового циклу.
- **Скасування**: Завжди підтримуйте передачу `CancellationToken` у довготривалі операції.

### ⚙️ 2.2. Запобігання блокуванню ігрового потоку
- Ніколи не викликайте `Thread.Sleep()` у головному потоці.
- Для затримок та періодичних дій використовуйте таймери гри:
  ```csharp
  // Правильно:
  api.Event.RegisterGameTickListener(OnGameTick, intervalMs);
  ```

### 🛠️ 2.3. Сучасні фічі C# 13
Активно використовуйте нові можливості мови для покращення читаності коду:
- **Primary Constructors** для лаконічного оголошення залежностей у класах систем.
- **Pattern Matching** для складних розгалужень логіки та перевірок сутностей.
- **Collection Expressions** (`[item1, item2]`) замість громіздких конструкцій масивів.

## 🎭 3. ПРОЕКТУВАННЯ СУТНОСТЕЙ (NPC & AI)

### 👣 3.1. Структура класів NPC
- Всі персонажі Середзем'я створюються на основі `EntityAgent`.
- Специфічний код для раси чи персонажа розміщується в `src/Entities/Humanoids/Entity[Name].cs`.
- Додаткові модулі поведінки підключаються через механізм **EntityBehavior** (C# або JSON).

### 🧠 3.2. Проектування ШІ (AI Tasks)
- Використовуйте стандартні та кастомні `AiTask` для побудови поведінки NPC.
- Пріоритети завдань у JSON повинні бути чітко збалансовані (наприклад, завдання атаки чи розмови мають вищий пріоритет, ніж блукання/wander).

## 🎨 4. КОНВЕНЦІЇ НАЗВ ТА РЕСУРСІВ (ASSETS)

- **ModID**: `lotr` (завжди в нижньому регістрі).
- **Префікси**: Кожен кастомний блок, предмет та сутність повинні мати префікс домену `lotr:` (наприклад, `lotr:lembas`, `lotr:gandalf`).
- **Шляхи ресурсів**: Ресурси мають строго відповідати структурі папок Vintage Story:
  - Моделі: `assets/lotr/shapes/entity/humanoid/[name].json`
  - Текстури: `assets/lotr/textures/entity/humanoid/[name].png`
  - Сутності: `assets/lotr/entities/humanoid/[name].json`

## ⏱️ 5. ПРОДУКТИВНІСТЬ ТА УПРАВЛІННЯ ПАМ'ЯТТЮ

Ігровий цикл працює на високій частоті кадрів (tickrate). Будь-який неоптимальний код у методах оновлення (`OnGameTick`, `Render`) швидко викличе лаги.

### 🗑️ 5.1. Уникайте виділення пам'яті (Zero Allocations in Hot Paths)
- Не створюйте нові об'єкти (класи, списки, масиви) всередині `OnGameTick` або циклів обробки кожної сутності.
- Перевикористовуйте вектори та математичні структури (`Vec3d`, `BlockPos`), кешуючи їх або передаючи за посиланням (`ref`).
- Використовуйте `BlockPos.Set(...)` замість створення нового `new BlockPos(...)`.

### ⛏️ 5.2. Робота зі світом
- Для масштабних змін блоків (наприклад, під час генерації світу чи споруд) використовуйте `IBlockAccessor` з типом `EnumMergeMode.Replace` або bulk-операції.
- Не викликайте оновлення освітлення для кожного блоку окремо — викликайте `IBlockAccessor.Commit()` після завершення пачки змін.

## 📝 6. ПРАВИЛА ВЗАЄМОДІЇ ТА ЛОГУВАННЯ

- **Debug Logging**: Використовуйте виключно `api.Logger.Notification("[LOTR] Повідомлення")` або відповідні методи логера (`Warning`, `Error`).
- **Спілкування з гравцем**: Інформаційні повідомлення для клієнта виводьте через `api.ShowChatMessage(...)`.
